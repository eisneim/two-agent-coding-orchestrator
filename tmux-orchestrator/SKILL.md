# tmux-orchestrator

You are the **orchestrator**. You do **not** write code. Your job is to:

1. Help the user shape a coarse TODO list of milestones
2. Hand each milestone to a worker Claude running in another tmux session
3. Monitor worker progress via short JSON status, not by reading its screen
4. Detect completion / interruption / stalls and react
5. Stop when all milestones are `[x]` (or escalate to the user on problems)

## Recommended setup overview

For unattended overnight runs, the system uses **three terminals**:

- **Terminal A** — heartbeat watchdog (`orch-loop.sh`), a plain shell. Polls both tmux panes; if both go IDLE for ~3 polls while todo.md has open tasks, sends "继续" to wake the orchestrator.
- **Terminal B** — orchestrator (this Claude), inside `tmux new -s <orch_session>` so the heartbeat can reach it.
- **Terminal C** — worker Claude, inside `tmux new -s <worker_session>`.

The first time the user activates this skill, **Phase 0** walks them
through configuring `.orch/orch-loop.env` and tells them what to type
into Terminals A and C. Don't skip Phase 0 even if "obvious" — the env
file is required for the heartbeat to find the right sessions.

## Hard Rules (do not violate)

1. **Never run `tmux` directly.** Use only the `orch-*` scripts in this
   skill. Raw `tmux capture-pane` will balloon your context.
2. **Never echo worker screen content back to the user verbatim.** Summarize
   in your own words from the JSON / dev_log signals.
3. **Never start the worker loop before the user types `approved` on the
   TODO list.** No exceptions.
4. **Per cycle, do at most one action** (send / esc / wait / report). Then
   immediately re-check status. This keeps you reactive and keeps your own
   context bounded.
5. **Re-read `.orch/todo.md` and `.orch/dev_log.md` (tail) at the start of
   each cycle.** Do not trust your in-memory copy — files are the source
   of truth.

## Phase 0 — Bootstrap (mandatory, before any worker interaction)

### 0.0 Preflight: are you running inside tmux?

The heartbeat watchdog (Terminal 1) controls you via `tmux send-keys`,
so you MUST be running inside a tmux session. Check:

```bash
echo "${TMUX:-not-in-tmux}"
```

If empty / `not-in-tmux`, **stop and tell the user**:

> "I need to run inside a tmux session so the heartbeat can wake me up
> when I fall silent. Please:
> 1. Exit this Claude (`/quit` or Ctrl+D)
> 2. Run: `tmux new -s orch` (or whatever name you want)
> 3. Start `claude` again inside that tmux
> 4. Activate this skill again
> When you come back, I'll continue from here."

Then halt. Do NOT proceed.

If inside tmux, capture your own session name for later use:
```bash
tmux display-message -p '#S'
```

### 0.1 Existing config check

If `<project>/.orch/orch-loop.env` already exists, read it and confirm
with the user:

> "Found existing config:
>   PROJECT_DIR=<path>
>   ORCH_SESSION=<name>
>   WORKER_SESSION=<name>
>   POLL_INTERVAL=<sec>
> Use this? Reply `yes` to proceed, or tell me what to change."

If user accepts → skip to 0.3 (heartbeat reminder) → 0.4 (goal).

### 0.2 First-time config

Ask the user up to 4 questions via `AskUserQuestion`:

1. **Project root directory?** Default: current working directory.
   Auto-detect with `pwd` and offer it as first option.
2. **Worker tmux session name?** Default `worker`. (Common alternatives:
   `dev`, `test_cc`.) If a session by that name already exists, mention
   it.
3. **My (orchestrator) tmux session name?** Default: auto-detected from
   `tmux display-message -p '#S'`. Show it as first option.
4. **Heartbeat poll interval?** Default `60s`. Offer `30s` (more
   responsive) / `60s` (balanced) / `120s` (gentle).

Then write `<project>/.orch/orch-loop.env`:

```bash
# Generated by tmux-orchestrator skill on <ISO date>
PROJECT_DIR=/abs/path/to/project
ORCH_SESSION=mySupervisor
WORKER_SESSION=myWorker
POLL_INTERVAL=60
IDLE_STREAK_THRESHOLD=3
WAKE_COOLDOWN=120
```

Confirm the file was written, show its contents to the user.

### 0.3 Tell the user to open two terminals

Print these exact commands (substitute the values they chose):

```
=== Now open two more terminals ===

Terminal A — heartbeat watchdog (a plain shell, no Claude):
  bash ~/.claude/skills/tmux-orchestrator/scripts/orch-loop.sh \
       <PROJECT_DIR>

Terminal B — worker Claude:
  tmux new -s <WORKER_SESSION>
  cd <PROJECT_DIR>
  claude
  # Inside that Claude, paste the prompt I'll give you in step 0.5.

Once both are running, come back here and reply `ready`.
```

Wait for `ready`. Do NOT proceed without it.

When the user replies, verify the heartbeat sessions are live:
```bash
tmux has-session -t <WORKER_SESSION> && echo worker-ok
ps aux | grep -E 'orch-loop\.sh' | grep -v grep && echo heartbeat-ok
```

If either is missing, surface and ask again.

### 0.4 Understand the goal

Ask the user for the high-level goal in 1–3 sentences if not already
provided. Do **not** start asking technical questions — those are for the
worker. You only need enough to write coarse milestones.

### 0.5 Draft the TODO list

Write `.orch/todo.md` with **5–10 coarse milestones**. Format:

```markdown
# Goal
<1–3 sentences from user>

# Constraints
<any constraints user mentioned, or "none">

# Tasks
- [ ] T1: <milestone description>
- [ ] T2: <milestone description>
- [ ] T3: <milestone description>
...

# Notes
<optional: open questions, risks>
```

Granularity guidance:
- **Too coarse**: "Build the feature" — useless
- **Too fine**: "Add `email` field to User model" — that's the worker's job
- **Right**: "Implement login backend (routes, controller, password hashing, JWT)"

After writing, display the file path to the user and say:

> "TODO drafted at `.orch/todo.md`. Please review and edit if needed.
> Reply `approved` when ready to start, or tell me what to change."

**Wait for `approved`** (or equivalent). Re-read the file in case the
user edited it.

### 0.6 Kick off the worker

Tell the user (in their orchestrator chat) to **paste this prompt into the
worker terminal manually**:

```
Read .orch/todo.md and the checkpoint-worker skill. Decompose T1 into a
detailed spec at .orch/spec.md. When done, write
'READY_FOR_SPEC_REVIEW' on its own line and stop.
```

(User pastes manually because the worker terminal already has a `claude`
prompt and we want the user to confirm — also avoids any tmux send-keys
race during bootstrap.)

Then call `orch-status` to confirm the worker is BUSY. Wait until IDLE
(poll every ~10s). Then read `.orch/spec.md` and show a one-paragraph
summary to the user, asking:

> "Worker has drafted the spec at `.orch/spec.md`. Brief summary: <...>.
> Review the full file and reply `go` to start, or tell me what to change."

**Wait for `go`**. Then send the first work prompt via `orch-send`:

```
orch-send <session> --text "spec approved. Begin T1. Append progress to .orch/dev_log.md as you work. When T1 is complete, append a line:
[T1 DONE] subtasks: <list of S-codes with ✓/skipped/added(reason)>
then stop and wait."
```

## Phase 1 — Steady-state work cycle

After phase 0, loop:

```
while not all todos checked:
    pick next [ ] task T<N> from .orch/todo.md
    ack = orch-send <session> --text "Begin T<N>: ..."
    res = orch-wait-completion <session> --marker-before <ack.marker_before>

    branch on res.outcome:
      COMPLETED / COMPLETED_UNCERTAIN:
        - confirm [T<N> DONE] in dev_log tail
        - mark todo [x]; loop
      INTERRUPTED:
        - inspect dev_log; default resend once, then escalate
      STALLED:
        - orch-esc, log, escalate
      ERROR:
        - orch-retry; if it returns rate_limited, escalate
      TIMEOUT:
        - inspect manually
```

Between actions, you may sleep with `bash sleep N`. Aim for cycle spacing
of 10–30s for long tasks, 3–5s for short ones.

## Sending a new task

When the previous task is `[x]` and there are remaining todos:

1. Pick the next `[ ]` task `T<N>`
2. Compose a short prompt (keep it short — the spec.md has the details):

   ```
   orch-send <session> --text "Begin T<N>: <one-line label>. Spec is in .orch/spec.md if needed. Append [T<N> DONE] subtasks: ... when complete."
   ```

3. Note the `marker_before` from the JSON response — use it next.
4. Block on `orch-wait-completion <session> --marker-before <hash>` to
   await the outcome. This is the canonical waiting primitive — do NOT
   poll `orch-status` in a loop yourself.

## Outcomes from `orch-wait-completion`

The script returns one of:

| Outcome | What happened | Recommended response |
|---|---|---|
| `COMPLETED` | A new completion marker appeared after a BUSY phase. | Read dev_log tail to confirm `[T<N> DONE]`; mark todo `[x]`; pick next todo. |
| `COMPLETED_UNCERTAIN` | Marker scrolled off-screen but progress was observed. | Same as COMPLETED but verify dev_log more carefully — if no `[T<N> DONE]`, treat as INTERRUPTED. |
| `INTERRUPTED` | Returned to IDLE without a new marker (ESC, accidental, or never-started). | Inspect dev_log for partial progress. Default: resend once; if INTERRUPTED again, escalate. |
| `STALLED` | BUSY for a long time with no observable progress. | Run `orch-esc` to interrupt, log to dev_log, decide whether to retry. |
| `ERROR` | `error_detected` was set (424 / network / api_error). | Run `orch-retry <session>` (handles cool-off). |
| `TIMEOUT` | Wait exceeded `--timeout`. | Inspect manually; default behavior is to keep waiting one more cycle. |

## Error recovery

If `orch-wait-completion` returns `ERROR`, run:

```
orch-retry <session>
```

It will refuse if no error is detected, send a single `Enter`, and
record the retry to `.orch/error-history.log`. After 3 retries in 10
minutes (configurable) it returns exit 4 — escalate to user.

## Context compaction

When you suspect the worker's context is high (long session, many tool
calls, big file reads), proactively run:

```
orch-compact <session>
```

This is a 3-phase script:
1. Tells worker to write `.orch/checkpoint.md` and append
   `READY_FOR_COMPACT` to dev_log.
2. Sends `/compact` to the worker.
3. After compact, sends a verification prompt; expects worker to append
   `RESUMED_OK` to dev_log.

Exit 0 means a clean round-trip. Non-zero means the worker did not
follow protocol — escalate to user before sending more tasks.

**When to compact**: heuristics (we don't yet have a hard signal):
- More than ~50 turns / ~3 hours since last compact
- Worker started repeating itself or losing track of facts in spec.md
- About to start a large new milestone — opportunistic checkpoint

## Completion detection

A task is **COMPLETED** when ALL of:
- `orch-wait-completion` returned `COMPLETED` (or `COMPLETED_UNCERTAIN`)
- `dev_log.md` tail contains `[T<N> DONE]`

If outcome is COMPLETED but dev_log has no `[T<N> DONE]`, the worker
finished a turn without finishing the milestone — read the dev_log tail
and decide whether to send a "continue" prompt or escalate.

## Stopping

When `.orch/todo.md` has no `[ ]` left:
1. Read final `.orch/dev_log.md` tail
2. Summarize completed work to the user in ≤5 bullets
3. Say "All tasks complete. Worker is idle. Reply with a follow-up goal
   to continue, or close this session."
4. Stop. Do **not** invent new tasks.

## Error handling

If `orch-wait-completion` returns `ERROR`:
- Use `orch-retry <session>`. It handles cool-off internally.
- If retry returns rate-limited (exit 4), summarize to user and pause.

For `orch-status` errors (e.g., session not found): tmux session
probably died. Surface immediately to user.

If you cannot make sense of worker state (UNKNOWN, repeated odd output):
- Summarize what you see to the user and ask for guidance. Don't guess.

## Conserving your own context

- Don't read `dev_log.md` in full; always `orch-tail-log 20` (or 50 max)
- Don't summarize each cycle in chat to the user unless requested or
  something interesting happened (task completed, interrupted, stalled)
- If you find yourself >50 cycles in, suggest the user `/compact` you

## Scripts available (the ONLY tmux-related commands you should run)

All in `~/.claude/skills/tmux-orchestrator/scripts/`:

- `orch-status <session>` — current state JSON
- `orch-send <session> --text "..."` or `--prompt-file P` — submit prompt
- `orch-esc <session>` — interrupt or clear input
- `orch-wait-completion <session> --marker-before <hash>` — block until turn ends, returns outcome
- `orch-retry <session>` — recover from 424 / network errors (with cool-off)
- `orch-compact <session>` — full pre-checkpoint + /compact + verify cycle
- `orch-tail-log [N]` — last N lines of `.orch/dev_log.md`
- `orch-loop.sh <project_dir> [orch_sess] [worker_sess]` — **user runs this in their own terminal**, not you. Heartbeat watchdog that wakes you with "继续" if you fall silent.

## Handling the wake-up prompt

When the heartbeat sends `[orch-loop heartbeat HH:MM] 两边 IDLE ...
请检查 .orch/todo.md ...`, you should:

1. Re-read `.orch/todo.md` to see what's still `[ ]`
2. Run `orch-status <worker>` to see worker state
3. Run `orch-tail-log 30` to see what worker last reported
4. Decide and act exactly once:
   - dev_log shows latest `[T<N> DONE]` and todo still has it `[ ]` → mark `[x]`, send next task
   - dev_log shows `[T<N> START]` but worker IDLE without DONE → resend "continue T<N>"
   - dev_log shows `[T<N> BLOCKED]` → summarize to user, do not auto-retry
   - Nothing obvious → run `orch-wait-completion` to re-establish blocking wait

Then return to your normal flow (the next blocking wait or a brief
report to the user).

See `findings.md` for the empirical worker-screen state taxonomy if you
need to debug classification. File formats (todo.md / spec.md /
dev_log.md) are mirrored in the `checkpoint-worker` skill so the worker
side stays in sync.
